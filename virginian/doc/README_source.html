00007 Introduction
00008 
00009 This is an experimental heterogeneous SQL database written to compare data
00010 processing on the CPU and NVIDIA GPUs. It was written by Peter Bakkum
00011 during the summer of 2010 at NEC Laboratories America in Princeton, New Jersey
00012 with several subsequent expansions. It shares no code with any other database
00013 system. A thanks goes to the Systems Architecture group at NEC, Srimat
00014 Chakradhar in particular, and the LAVA Lab group at UVa, Prof Kevin Skadron in
00015 particular.
00016 
00017 The purpose of this database is to test and demonstrate several novel ideas
00018 relating to RDBMS-like computation on the GPU. Thus, all queries can be executed
00019 on the CPU with a single thread, the CPU with multiple threads, or the GPU.
00020 Chief among the ideas demonstrated here is the data structure used both for data
00021 storage and execution across both platforms, the Tablet, around which the entire
00022 database has been constructed.
00023 
00024 This is a "SQL Database" in that it has a SQL interface and is capable of
00025 performing several types of queries on non-volatile data, but is far from
00026 supporting the full SQL standard. It is not suitable for production, but has
00027 been written to be small and make it easy to experiment and compare execution on
00028 the CPU and GPU.
00029 
00030 This code was developed on Linux 2.6 machines and tested with NVIDIA Tesla
00031 C1060, Geforce GTX 570, and Tesla C2050 cards on CUDA 3.0 and above. Compiling
00032 will at the very least probably involve tweaking the project makefile, and might
00033 involve changing some code. The code involves system calls to unix libraries, so
00034 compiling on anything other than Linux may be harder, and compiling on Windows
00035 may require some major changes, though I've never tried it. As stated, this is
00036 an experimental project and it will be difficult to use if you are unfamiliar
00037 with C and CUDA.
00038 
00039 Documentation generated by Doxygen is provided in the doc/ folder. It can be
00040 viewed by opening html/index.html with a web browser or by running make in the
00041 latex/ directory to generate a pdf of the documentation.

00056 Compilation
00057 
00058 Initial compilation should be done by running make in the project root
00059 directory. This will download and compile the GNU Scientific Library, which
00060 is used for generating random numbers for test databases. It will then create
00061 a 1,000,000 row test database and call make in the src/ directory, which
00062 will make all of the project source files and run a test to compare the
00063 running times of queries for single and multicore CPU execution and GPU
00064 execution. These tests can be run again just by executing make in the src/
00065 directory.
00066 
00067 There are 3 modes of compilation: debug, gccrelease, and iccrelease. The
00068 gccrelease settings enable optimization, and the iccrelease settings compile
00069 with optimization using icc. Timing results should only be reported from
00070 release settings. The compilation mode is set in the Makefile in the src/
00071 directory.
00072 
00073 Before compiling you should look in src/Makefile and tweak the settings for your
00074 machine.

00109 Using the Virginian API
00110 
00111 This project's interface is well documented but I haven't taken the step of
00112 dividing the user-facing and internal functionality, so the documentation covers
00113 both. Using the code externally should be fairly straightforward. The
00114 compilation and testing process outputs an archive file to the lib/ directory
00115 that can be compiled with another C/C++ project. The files you need are
00116 lib/virginian.a and lib/virginian.h. The example/ directory contains a very
00117 simple project useful for getting started using this code.

00119 Compile-time Options
00120 
00121 There are several macros that change the program at compile time. They are
00122 added by editing the CUSTOM_FLAGS variable of the Makefile in the src/
00123 folder. The format should be -D MACRO_NAME.
00124 
00125 - VIRG_DEBUG
00126   Enables debugging code, such as printing out the higher-level opcodes as
00127   they execute and initializing memory areas to 0xDEADBEEF.
00128 
00129 - VIRG_DEBUG_SLOTS
00130   Enables tablet slot information printing to stdout. This prints out a
00131   listing of the tablet ids and locks whenever a lock is acquired or
00132   released.
00133 
00134 - VIRG_NOPINNED
00135   Do not declare the main-memory tablet slots as pinned. This switches the
00136   allocation to a malloc() rather than a cudaMallocHost(). This means that
00137   GPU execution cannot use streaming or mapped memory.
00138 
00139 - VIRG_NOTWOSTEP
00140   Do not buffer results temporarily in GPU global memory before writing them
00141   to main memory. This two-step copy operation makes mapped memory on the
00142   C1060 faster by an order of magnitude but should be disabled for ION boards
00143   because there is no distinction between GPU memory and main memory.

00146 Opcodes
00147 
00148 These opcodes are the steps into which a query is broken down. Each opcode
00149 has four arguments, though few opcodes use all four. The first three are
00150 integer values, while the fourth can be any type.
00151 
00152 In general the first two opcodes are used to refer to virtual machine
00153 registers and the third is used to refer to a program counter value, such as
00154 the end of the parallel section. These are just conventions, however, and not
00155 all opcodes are organized in this way.
00156 
00157 These opcodes are loosely based on the SQLite opcodes but have a number of
00158 subtle semantic differences. The SQLite opcodes can be seen at
00159 http://sqlite.org/opcode.html. The changes have been made both to better
00160 support heterogeneous execution and to make things a little simpler. The
00161 biggest change is that even though the virtual machine operating on each row
00162 can diverge and have an independent program counter, that program counter can
00163 never decrease. In other words, execution over the data-parallel segment
00164 proceeds as a waterfall, so opcodes can only jump ahead.
00165 
00166 Higher-Level Opcodes
00167 
00168 These opcodes control setting up query execution and launching the
00169 data-parallel segment. These opcodes do not handle row data directly and are
00170 implemented in vm/execute.c. They are listed in alphabetical order.
00171 
00172 - Finish [], [], [], []
00173   This opcode marks the end of the data-parallel segment of query execution.
00174   Its purpose is to provide a jump location for the higher-level virtual
00175   machine and to clean up the remaining tablet locks of the data-parallel
00176   segment.
00177 
00178 - Parallel [], [], [end of parallel section], []
00179   Begins the data-parallel segment of query execution. Depending on the
00180   configuration settings, this either launches a single-core, multi-core, or
00181   GPU virtual machine that treats every row as independent in its processing.
00182   The third argument refers to the opcode marking the end of the parallel
00183   segment, which should always be Finish.
00184 
00185 - ResultColumn [column type], [], [], [column name]
00186   Prepares the result tablet to receive a certain column of data. The type is
00187   one of the integer enumerations of data types, such as VIRG_INT, and the
00188   name is a character pointer to a location in memory of a c string
00189   containing the result column names. Note that only the semantics to call
00190   this opcode once are currently implemented.
00191 
00192 - Table [table id], [], [], []
00193   Opens a handle on a table for execution. Required when beginning execution.
00194

00196 Lower-Level Opcodes
00197 
00198 These opcodes are used to work directly with the data of a table, and can be
00199 independently executed for each row. They are specifically designed to be
00200 platform independent, meaning they are used for both CPU and GPU execution.
00201 Intermediate data during this lower-level step is stored in the virtual
00202 machine registers, which are unions that allow the virtual machine to
00203 temporarily store any variable type. They should not be confused with the
00204 notion of the CUDA register space, and in fact are stored in local memory on
00205 the GPU.
00206 
00207 The four arguments are referred to as p1, p2, p3, and p4. Register locations
00208 are referred to as a reg[] array. Thus, reg[p1] refers to the p1-th virtual
00209 machine register.
00210 
00211 One of the virtual machine elements associated with each row is the concept
00212 of validity. The SELECT operation is just a filter, so we determine which
00213 rows are valid and invalid, and the ResultRow opcode only outputs those rows
00214 which are valid. This is different from the SQLite semantics, where only
00215 valid rows execute this opcode, and this change was intended to better
00216 support coordination among threads on the GPU and within the SIMD block on
00217 the GPU.
00218 
00219 - Add [destination register], [source register 1], [source register 2], []
00220   Executes the operation reg[p1] = reg[p2] + reg[p3].
00221 
00222 - And [comparison register 1], [comparison register 2], [jump location],
00223   [validity]
00224   If (reg[p1] && reg[p2]) evaluates to true, then the program counter
00225   associated with this row jumps to p3 and the rows validity is set to the
00226   integer value of p4.
00227 
00228 - Cast [target type], [register], [], []
00229   Casts the value in register p2 to the type denoted by p1, such as VIRG_INT.
00230 
00231 - Column [destination register], [source column], [], []
00232   Loads the value from from the p2-th fixed column of the current table and
00233   places it in the p1-th register.
00234 
00235 - Converge [], [], [], []
00236   Marks the end of the data-parallel query execution segment. This opcode
00237   causes the lower-level virtual machine to exit and return control to the
00238   higher-level virtual machinemachine.
00239 
00240 - Div [destination register], [source register 1], [source register 2], []
00241   Executes the operation reg[p1] = reg[p2] / reg[p3].
00242 
00243 - Eq [comparison register 1], [comparison register 2], [jump location],
00244   [validity]
00245   If (reg[p1] == reg[p2]) evaluates to true, then the program counter
00246   associated with this row jumps to p3 and the rows validity is set to the
00247   integer value of p4.
00248 
00249 - Float [destination register], [], [], [constant]
00250   Sets the value of reg[p1] to the float value of p4.
00251 
00252 - Ge [comparison register 1], [comparison register 2], [jump location],
00253   [validity]
00254   If (reg[p1] >= reg[p2]) evaluates to true, then the program counter
00255   associated with this row jumps to p3 and the rows validity is set to the
00256   integer value of p4.
00257 
00258 - Gt [comparison register 1], [comparison register 2], [jump location],
00259   [validity]
00260   If (reg[p1] > reg[p2]) evaluates to true, then the program counter
00261   associated with this row jumps to p3 and the rows validity is set to the
00262   integer value of p4.
00263 
00264 - Integer [destination register], [constant], [], []
00265   Sets the value of reg[p1] to p2, which is an integer value.
00266 
00267 - Invalid [], [], [], []
00268   Invalidates all rows that have this opcode executed, so the only rows that
00269   are output by ResultRow are those that jump over this opcode.
00270 
00271 - Le [comparison register 1], [comparison register 2], [jump location],
00272   [validity]
00273   If (reg[p1] <= reg[p2]) evaluates to true, then the program counter
00274   associated with this row jumps to p3 and the rows validity is set to the
00275   integer value of p4.
00276 
00277 - Lt [comparison register 1], [comparison register 2], [jump location],
00278   [validity]
00279   If (reg[p1] < reg[p2]) evaluates to true, then the program counter
00280   associated with this row jumps to p3 and the rows validity is set to the
00281   integer value of p4.
00282 
00283 - Mul [destination register], [source register 1], [source register 2], []
00284   Executes the operation reg[p1] = reg[p2] * reg[p3].
00285 
00286 - Neq [comparison register 1], [comparison register 2], [jump location],
00287   [validity]
00288   If (reg[p1] != reg[p2]) evaluates to true, then the program counter
00289   associated with this row jumps to p3 and the rows validity is set to the
00290   integer value of p4.
00291 
00292 - Not [test register], [], [jump location], [validity]
00293   If reg[p1] evaluates to true, then the program counter
00294   associated with this row jumps to p3 and the rows validity is set to the
00295   integer value of p4.
00296 
00297 - Or [comparison register 1], [comparison register 2], [jump location],
00298   [validity]
00299   If (reg[p1] || reg[p2]) evaluates to true, then the program counter
00300   associated with this row jumps to p3 and the rows validity is set to the
00301   integer value of p4.
00302 
00303 - Result [start register], [result columns], [], []
00304   If the current row is still valid then output the registers from reg[p1] to
00305   reg[p1 + p2] as a row of fixed-size results. The CPU virtual machine
00306   attempts to efficiently group the copies of a column in adjacent rows,
00307   while the GPU coalesces these writes after an atomic scatter operation
00308   conducted in two-steps in shared and global memory.
00309 
00310 - Rowid [destination register], [], [], []
00311   Loads the value of the current row's primary key and places it in reg[p1]
00312 
00313 - Sub [destination register], [source register 1], [source register 2], []
00314   Executes the operation reg[p1] = reg[p2] - reg[p3].
